MFA Project - 365 & VPN

Securing access to cloud services and network infrastructure via multi-factor authentication.

Project Objective:

During this project spread over a week, the goal is to implement an MFA solution. This solution will deepen security at the client's site.

Client Situation:

The company has an IT infrastructure notably composed of a WatchGuard firewall, which allows connection via VPN, as well as an Active Directory server synchronized with the Microsoft 365 cloud. The client wants its users to have access to the Office 365 suite and the VPN managed according to the implementation of MFA as well as their location.

Link with the cyber track: This project fits perfectly into my track because it allows me to see the implementation from A to Z of an additional security layer within a company. Here, MFA is used to prevent the compromise of the IT service during a password leak or phishing.

Environment and Tools

Microsoft Entra ID

Administration center used for managing cloud identities, creating conditional accesses, and hybrid synchronization.

Active Directory

Local directory service synchronized with Entra ID via Connect Sync for managing hybrid accounts and groups.

Microsoft 365

Cloud application suite where all access is filtered and secured by the deployment of our MFA.

WatchGuard VPN

Network firewall configured to delegate its teleworkers' multi-factor authentication via SAML integration.

Authentication Methods

MFA (Multi-Factor Authentication) is a method that adds an extra layer of security in the connection. It is useful when a user has had their password stolen, for example. This method materializes by sending a notification to the user after entering their password.

During this project, we set up two means of two-factor authentication:

Microsoft Authenticator:

Authenticator is an application created by Microsoft that allows, in particular, connection to the Office 365 suite. When entering the password, a pop-up appears on the PC with a two-digit number. The user can then enter this code on their phone to connect.

Physical Token (TOTP):

Token C202 from token2. This time, upon connection, a 6-digit code is requested. This is generated by the token using the SHA1 hash of its seed (unique secret key) and a time factor.

Logic and Policy Details

Before implementing our conditional accesses in Entra ID, it was necessary to summarize the logic of the rights with tables. Indeed, a person can only be in one of these groups at a time and the required MFA method strongly depends on the profile.

Procedure - MFA 365 Part

1 - Group Creation on the AD and Entra

For each of the possibilities defined above, we created a security group in the AD GS_..._AD which groups the users to whom these policies apply.

In parallel, we created their equivalent in Microsoft Entra for "cloud only" accounts (which are not defined in the AD), such as certain administrator accounts or generic accounts like shared mailboxes.

Note: A person can only be in one of these groups at a time.

2 - Creation of Authentication Strengths

We define here the different connection means for the users. If we take the categories, we configured 4 methods:

  • Password only
  • Password and third-party software token
  • Password and physical token
  • Password and Microsoft Authenticator

3 - Creation of Named Locations

So that the connection conditions apply based on location, we created named locations, defining areas to which the policies will then be applied.

We were able to define these points in two ways:

  • For countries, we chose country by country.
  • For locations like the agency and the two local sites, we specified a precise list of public IPs.

4 - Creation of Policies

We will assign authentication methods to the different groups created, as well as times before re-authentication (for example, someone working on-site will have to reconnect via Authenticator every 3 months, whereas someone working outside France will have to reconnect every 7 days).

Policy creation logic:

If a user falls into two policies at the same time, the most restrictive one applies. Knowing this, we first created the policies that block the connection:

  • BLOQUER_NO_GROUP: if the user is in none of the groups, their authentication is necessarily blocked.
  • BLOQUER_MDP_UNIQUEMENT_EXT: if the user only has a password without MFA, they are blocked everywhere except on-site.
  • BLOQUER_JETONS_PHYSIQUE_EXT / BLOQUER_JETONS_LOGICIEL_EXT: If the user has a physical token or a third-party software token, they are blocked if connecting outside of France.
  • BLOQUER_MSAUTH_FRANCE: Same thing for the Authenticator version.
  • BLOQUER_MSAUTH_MONDE: If the user connects from another unauthorized country, they are necessarily blocked.

Access grant policies:

Next, we define our policies for cases where the user is allowed to connect:

  • AUTH_MDP_UNIQUEMENT_30J
  • AUTH_MDP_PLUS_JETON_LOGICIEL_90J
  • AUTH_MDP_PLUS_JETON_PHYSIQUE_90J
  • AUTH_MDP_PLUS_MSAUTH_7J
  • AUTH_MDP_PLUS_MSAUTH_30J
  • AUTH_MDP_PLUS_MSAUTH_90J

For example, if a user from the GS_MDP_PLUS_MSAUTH_MONDE group connects from Germany, an authorized country, a periodic re-authentication of 7 days is applied to them and connection via Authenticator is forced.

Once our policies are defined and applied, we can have the users install the application.

Part Two: Implementing MFA for the VPN

Users connecting to the VPN must be part of the SSLVPN-Users group in the AD.

For the connection, we were able to delegate authentication to Microsoft and thus apply our previously defined policies. To set up this connection mode, we went through the addition of a SAML authentication server. To do this, in Entra -> Enterprise Applications we added Firebox Authentication Portal SAML.

In the Single sign-on section, we configured our VPN, notably with a token signing certificate retrieved from the VPN administration page vpn.domain.com/auth/saml. Once this link is configured, we were able to manage the user section.

User Side:

This delegation looks like this: By selecting “Authenticate with SAML”, the user will see a classic Microsoft login page appear, followed by a code for the MFA as usual.

Note: need to update the VPN client for connection via SAML.

Other Completed Tasks

Email Security (DMARC, SPF, DKIM)

We implemented a DMARC policy allowing the sender to indicate that their emails are protected. SPF serves as a "directory" of sending servers and DKIM as a signature on emails. These records are stored in the DNS.

Microsoft 365 Audit

We conducted a global audit: reviewing users and admin accounts, verifying licenses and Active Directory / Entra ID synchronization, as well as analyzing the privacy issues of public groups.

Powershell Automation

We used Powershell scripts to perform exports of the Active Directory to an Excel spreadsheet to ensure precise monitoring of accounts (Hybrid or Cloud only) and security group memberships during the deployment.

Technical Documentation

We created numerous materials: detailed policy diagrams (.drawio), user review guides, physical token assignment manuals, and procedures for resolving synchronization problems.

Summary of Problems Encountered

  • The logic of the policies requires strict discipline because the most restrictive rule always applies. A configuration error could have locked out access for the entire company.
  • Outlook configuration problems related to different software versions and user habits (difference in handling between Modern Authentication and the Legacy version).
  • Difficulties in registering and synchronizing physical tokens in the cloud, requiring strict adherence to Microsoft's formats.
